# CONSTRAINTS

## 目标

本文档用于记录 `dotbot-go` 当前阶段的关键取舍, 说明为什么项目主动不做某些能力.

这些约束不是权宜之计, 而是为了让第一版保持边界清晰, 行为稳定, 实现和维护成本可控.

## 为什么不用 YAML

- `dotbot-go` 当前采用的是固定 section 的声明式配置模型, 与 TOML 更匹配
- TOML 结构更稳定, 对缩进不敏感, 更适合做严格校验
- TOML 对 `default`, `create`, `clean`, `[[link]]` 这种结构表达更分明
- 项目当前更看重配置语义清楚和类型稳定, 而不是最大限度兼容上游习惯

## 为什么不用任务列表模型

- `dotbot-go` 希望用户描述目标状态, 而不是自己编排执行步骤
- 顺序语义会把配置文件逐渐推向脚本化, 增加理解成本
- 声明式模型更适合做 schema 校验, 默认值归并, dry-run 展示和错误定位
- 内部执行顺序仍然存在, 但由工具固定为 `create -> link -> clean`, 不暴露为用户能力

## 为什么不支持 shell

- shell 很容易把项目边界扩展成通用任务执行器
- 一旦支持 shell, 就需要处理执行环境, 输出控制, 幂等性, 错误语义等额外复杂度
- 真有命令执行需求时, 用户完全可以在仓库里维护独立脚本
- `dotbot-go` 当前只专注于 dotfiles 管理本身

## 为什么不支持 plugin

- plugin 会显著增加扩展契约, 生命周期, 兼容性和文档负担
- 当前项目目标不是构建生态平台, 而是先把核心 dotfiles 管理能力做好
- 固定能力集合更利于严格校验, 统一行为和输出风格
- 第一版优先保证简单, 稳定, 可解释

## 为什么只支持 symlink

- symlink 是 dotfiles 场景最常见, 也最容易理解的链接方式
- hardlink 会引入额外平台差异和行为复杂度, 但对当前目标收益有限
- 只支持 symlink 可以让 `relative`, `relink`, `force` 的语义更集中

## 为什么不支持环境变量展开

- `~` 已经足够覆盖最常见的 home 目录定位需求
- 环境变量展开会增加平台差异和额外解释规则
- 当前只保留 `~` 这一种显式锚点, 有利于路径语义稳定

## 为什么失败即停, 但不承诺回滚

- 回滚语义看起来安全, 但真正做到可靠回滚会显著增加实现复杂度
- 覆盖普通文件, 删除死链, 创建目录等行为, 很难完整恢复执行前状态
- 当前项目选择更真实的策略: 失败即停, `force = true` 时由用户承担覆盖风险
- 这比提供一套不完全可靠的“回滚承诺”更容易解释和维护
