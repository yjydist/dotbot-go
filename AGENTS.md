# AGENTS

## 目标

本文件用于约束 `dotbot-go` 在编码阶段的默认工作方式.

原则上, 先遵循本文件, 再结合 `DESIGN.md` 和 `TODO.md` 推进实现.

## 交流与输出

- 默认使用中文交流
- 标点符号统一使用半角
- 对外文档, README, 示例, 帮助文案默认使用中文
- CLI 帮助文案和错误提示优先简洁直接
- 每轮完成后按 `改动 + 验证 + 风险` 的结构汇报

## 开发流程

- 默认采用 `先设计, 后实现` 的方式推进
- 如果实现过程中发现需要调整既定设计或配置 schema, 必须先讨论, 不能直接改
- 默认采用最小改动原则, 不顺手扩大重构范围
- 如果发现死代码, 旧草稿, 或可清理内容, 先标记并说明, 不默认顺手删除
- 遇到未预期问题时, 先定位并收集足够信息, 再带结论和建议汇报

## 代码风格

- 可读性优先于性能优化
- 命名优先语义清晰, 不追求过短
- 函数尽量小而清晰, 职责单一
- 文件不要过大, 优先按职责拆分
- 代码注释采用适量策略, 仅为关键逻辑, 非直观分支, 边界处理补充说明
- 默认尽量少导出标识符, 优先缩小 API 暴露面

## Go 项目结构

- 采用标准清晰分层
- 优先使用 `cmd + internal` 结构
- 早期包划分也要保持职责清楚, 但不要过度抽象
- 默认按职责拆包, 例如配置解析, 执行器, 链接处理, 清理处理, 输出处理

## 错误处理

- 优先显式返回 `error`
- 不要过早引入复杂错误框架
- 配置解析和运行时错误要清楚区分
- 错误信息要尽量保留字段路径, 配置段, 目标路径或源路径
- 对用户可见的错误信息要便于定位和修复

## 配置与兼容策略

- 配置解析采用严格校验
- 未知字段直接报错
- 非法值直接报错
- 不做宽松兼容和隐式兜底
- 实现时必须遵循 `DESIGN.md` 中已经定稿的 schema 和规则

## 测试策略

- 先单测为主
- Go 测试优先使用表驱动测试
- 一个功能达到可提交状态前, 至少应覆盖本次改动的核心逻辑
- 优先补关键路径测试, 再补边界测试和快照类测试
- 测试范围以 `TODO.md` 中当前阶段目标为准

## 格式化与检查

- 默认执行自动格式化
- 提交前至少运行格式化和核心测试
- 基础检查默认包括格式化和基础静态检查
- 如果某次改动没有完成核心验证, 必须在汇报中明确说明

## 依赖策略

- 新依赖采用实用优先原则
- 但新增依赖前应先判断标准库是否足够
- 依赖版本优先选择稳定版本
- 引入依赖时要说明用途和收益

## Git Workflow

- 单人项目默认直接在 `main` 分支开发
- 每完成一个独立小功能或小修复就提交一次
- commit message 使用中文简短描述
- 默认保持线性历史
- 默认避免使用 `rebase`, `amend`, `reset --hard` 等历史改写或破坏性操作
- 若确实需要高风险 Git 操作, 先说明风险和影响
- 一个功能完成后, 优先同步更新相关文档, 再提交

## 风险操作规则

- 涉及覆盖文件, 删除死链, `force` 行为时, 先明确说明风险再执行
- 即使行为符合设计, 也要在执行前向用户说明影响
- `force = true` 表示用户承担覆盖风险, 但编码实现中仍应尽量把风险提示做清楚
- `target` 虽然允许相对路径, 但文档和实现提示中默认不鼓励, 优先推荐 `~` 或绝对路径

## TODO 维护规则

- 编码阶段始终维护 `TODO.md`
- 每次开始一轮实现前, 先更新 `TODO.md`
- 每完成一个任务后, 立即更新状态
- 任务粒度保持中等, 一般一个功能拆成 3 到 7 个可执行项
- `TODO.md` 用于记录当前编码进度, 不仅仅记录远期规划

## 文档同步

- 实现过程中, 文档采用边实现边同步的方式维护
- 文档组织默认采用 `README 精简 + DESIGN 细化` 的方式
- 规则变化时及时更新 `README.md` 和 `DESIGN.md`
- 行为变化时及时更新示例和 dry-run 输出样例

## 输出与平台细节

- 终端输出默认允许颜色, 并提供 `--no-color` 关闭
- `clean` 默认采用保守清理策略

## 默认执行原则

- 优先真实, 清晰, 可维护
- 优先最小改动, 不做无关扩展
- 优先让代码和行为容易解释
- 如果设计和实现发生冲突, 先停下来讨论, 再继续推进
